# Firestore â€” data model and indexes

Reference for the `biwenger-tools` Firestore database. Keep it up to date by
hand when collections, fields, or indexes change â€” nothing is autogenerated.

- **GCP project:** `biwenger-tools`
- **Database:** `(default)`, Native mode, regional **`europe-southwest1`**
  (free tier, co-located with Cloud Run).
- **Auth:** Application Default Credentials (ADC).
  - In Cloud Run the compute SA is picked up automatically.
  - Locally: `gcloud auth application-default login` once.

---

## Collections

```
comunicados/{season}/messages/{id_hash}
participacion/{season}/authors/{autor}
clausulazos/{season}/transfers/{content_hash}
tabla_justicia/{season}/teams/{equipo}
palmares/{temporada}
```

There is an intermediate "season" document (`comunicados/{season}`,
`participacion/{season}`, etc.) that is empty â€” it only exists as a
container for the subcollection. Firestore creates it implicitly.

The Python models that serialize/deserialize these documents live in
`core/domain/models.py` with `to_firestore()` / `from_firestore()` methods.

## Schemas

### `comunicados/{season}/messages/{id_hash}` â€” `LeagueMessage`

League messages, fun facts, chronicles, and loans from Biwenger's board.
The category (`comunicado`, `dato`, `cronica`, `cesion`) is a field, not
part of the path â€” the same document can move between sections of the
web app without being relocated.

| Field       | Type                       | Notes |
|-------------|----------------------------|-------|
| `fecha`     | timestamp                  | Serialized from `dd-MM-YYYY HH:mm:ss` |
| `autor`     | string                     | Manager name |
| `titulo`    | string                     | |
| `contenido` | string                     | HTML; sanitized on read by the web (`safe_html`) |
| `categoria` | string                     | Enum: `comunicado` / `dato` / `cronica` / `cesion` |

**Doc id:** SHA-256 hex of the original `(date + content)`, computed in
`scraper_job/main.py::_process_new_messages`. Deterministic â€” the same
message always maps to the same id, which makes the dual-write idempotent.

### `participacion/{season}/authors/{autor}` â€” `Participation`

Aggregated per manager: which messages they posted and in which category.
The scraper recomputes this every run (no incrementing â€” full rewrite).

| Field         | Type                | Notes |
|---------------|---------------------|-------|
| `comunicados` | array&lt;string&gt; | IDs (`id_hash`) of the league messages signed |
| `datos`       | array&lt;string&gt; | IDs of the fun facts |
| `cesiones`    | array&lt;string&gt; | |
| `cronicas`    | array&lt;string&gt; | |
| `total`       | int                 | Derived (sum of lengths); stored on write so we can `order_by("total")` server-side |

**Doc id:** the manager's name (`autor`).

### `clausulazos/{season}/transfers/{content_hash}` â€” `Clausulazo`

Player transfers executed via release clause.

| Field               | Type       | Notes |
|---------------------|------------|-------|
| `fecha`             | timestamp  | Serialized from `dd-MM-YYYY HH:mm` (no seconds) |
| `jugador`           | string     | |
| `equipo_vendedor`   | string     | |
| `equipo_comprador`  | string     | |
| `precio`            | int        | Euros (`6475000` = 6.4M); same as Biwenger's API |

**Doc id:** SHA-256[:20] of `fecha|jugador|equipo_vendedor|equipo_comprador|precio`
(`scraper_job/main.py::_clausulazo_doc_id` and
`scripts/backfill_firestore.py::_clausulazo_id`). Deterministic â€” the
scraper and the backfill must agree so rewrites are no-ops.

### `tabla_justicia/{season}/teams/{equipo}` â€” `JusticeEntry`

Summary of "attacks" (clausulazos made/received) per team.

| Field              | Type                            | Notes |
|--------------------|---------------------------------|-------|
| `total_hechos`     | int                             | Number of clausulazos this team made |
| `total_recibidos`  | int                             | |
| `punto_de_mira`    | string                          | The team it targets the most |
| `mayor_agresor`    | string                          | The team that targets it the most |
| `hechos`           | array&lt;map&lt;string,int&gt;&gt; | `[{team, count}, ...]` desc order |
| `recibidos`        | array&lt;map&lt;string,int&gt;&gt; | |

**Doc id:** the team's name (`equipo`). The placeholder team for managers
who have left is called `Usuario` (Biwenger's convention).

### `palmares/{temporada}` â€” `Palmares`

Historical honours. One document per season (`24-25`, `23-24`, ...).

| Field              | Type                | Notes |
|--------------------|---------------------|-------|
| `campeon`          | string              | |
| `subcampeon`       | string              | |
| `tercero`          | string              | |
| `farolillo`        | string              | Last placed (pays fine) |
| `puntuacion`       | string              | Champion's "score (decisive)" |
| `record_puntos`    | string              | e.g. `"112 @fabio"` |
| `jornadas_ganadas` | string              | |
| `multas`           | array&lt;string&gt; | Who pays fines (can be >1) |

**Doc id:** the season (`24-25`, etc.). Alphabetical DESC equals
chronological DESC, so `order_by("__name__")` returns e.g.
`["25-26", "24-25", "23-24"]`.

## Indexes

Firestore auto-indexes **every root field** and **arrays** (membership) in
both ascending and descending direction. It does **not** auto-index field
combinations â€” those are **composite indexes** and must be declared.

### Auto (nothing to do)

Every "order by a single field" query works out of the box:

| Collection                                | Query                                          |
|-------------------------------------------|------------------------------------------------|
| `participacion/{season}/authors`          | `order_by("total", DESCENDING)`                |
| `clausulazos/{season}/transfers`          | `order_by("fecha", DESCENDING)`                |
| `tabla_justicia/{season}/teams`           | `order_by("total_hechos", DESCENDING)`         |

**Caveat:** Firestore does **not** auto-index `__name__` in DESCENDING
order. If you need to order a collection by its document id descending,
either create a single-field DESC index for `__name__` on that
collection, or â€” when the collection is tiny â€” sort in Python after a
plain `.stream()` (that's what `get_palmares()` does).

### Composite (declared)

| Collection group | Scope      | Fields                              | Purpose |
|------------------|------------|-------------------------------------|---------|
| `messages`       | COLLECTION | `categoria` ASC, `fecha` DESC       | Paginate and filter comunicados / salseo (`get_messages_by_category`) without scanning all ~2,800 messages |

`queryScope: COLLECTION` makes the index apply to **every subcollection
named `messages`** (one per season). Adding a new season needs no
additional index.

### Where they live

`firestore.indexes.json` at the repo root is the declarative source of
truth.

### Create / update indexes

```bash
# Create the messages index (idempotent â€” soft-fails if it already exists)
gcloud firestore indexes composite create \
  --collection-group=messages \
  --query-scope=COLLECTION \
  --field-config=field-path=categoria,order=ascending \
  --field-config=field-path=fecha,order=descending \
  --project=biwenger-tools

# List current indexes
gcloud firestore indexes composite list --project=biwenger-tools

# Delete one by id (rare)
gcloud firestore indexes composite delete <id> --project=biwenger-tools
```

When a query needs an index that does not exist, Firestore returns
`FAILED_PRECONDITION` with a direct link to the console to create it.
Building takes seconds to minutes depending on collection size.

## Where things are read and written

| Operation | File | Notes |
|-----------|------|-------|
| Web reads | `packages/biwenger_tools/web/repository.py` | One query per function, inline â€” no generic abstraction |
| Writes (scraper) | `packages/biwenger_tools/scraper_job/main.py` | Dual-write CSV + Firestore via `_write_to_firestore` |
| Backfill (one-shot) | `scripts/backfill_firestore.py` | Wipe + bulk-write from existing CSVs |
| SDK | `core/sdk/firestore.py` | Generic helpers only: `get_client`, `list_documents`, `set_document`, `query`, `count`, `batch_write`, `delete_collection` |

## Reads per page

With the current model + indexes, a normal visit costs:

| Page | Reads |
|------|-------|
| `/<season>/` (comunicados) | 1 count + 7 page = **~8** |
| `/<season>/` with active search box | +~530 (first time per session, via `comunicados/search-data`) |
| `/<season>/salseo` | ~600 (datos + cronicas + clausulazos + tabla_justicia) |
| `/<season>/participacion` | ~7 |
| `/<season>/mercado` | ~111 |
| `/palmares` | ~3 |

Free tier: 50,000 reads/day and 20,000 writes/day. Plenty of headroom for
normal usage.
